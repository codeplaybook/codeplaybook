# .codeplaybook/ Format Specification

The `.codeplaybook/` directory is the agent-agnostic source of truth for a project's coding standards and commands. It is generated by the `onboard` and `prescribe` workflows, and consumed by the `sync` workflow to deploy to specific coding agents.

## Directory Structure

```
.codeplaybook/
├── standards/
│   ├── {slug}.md              # One file per standard
│   └── ...
├── commands/
│   ├── {slug}.md              # One file per command
│   └── ...
└── _drafts/                   # Generated but not yet deployed
    ├── standards/
    └── commands/
```

## Naming Convention

All files use slugified names with a `codeplaybook-` prefix:
- Lowercase, hyphen-separated
- Example: "Test Data Factories" → `codeplaybook-test-data-factories.md`
- Collision handling: if a slug already exists, append `-2`, `-3`, etc.

## Standard Format

```markdown
# {Standard Name}

{Description — what this standard enforces and why it matters.
Reference evidence found during analysis when applicable.}

## Scope

{Where this standard applies. Be specific:
- "TypeScript files in src/"
- "React components"
- "All test files"
- "Python files in the services/ directory"}

## Rules

* {Rule 1 — start with imperative verb, max ~25 words}
* {Rule 2}
* {Rule 3}
```

### Field Details

| Field | Required | Description |
|-------|----------|-------------|
| Name (H1) | Yes | Human-readable standard name |
| Description | Yes | What the standard covers and why. First sentence used as summary in agent config. |
| Scope | Yes | Where rules apply. Used by agent adapters to generate file-matching patterns (globs). |
| Rules | Yes | Concrete, actionable rules. Each starts with an imperative verb. |

## Command Format

```markdown
# {Command Name}

{What the command does, why it's useful, and when it's relevant.}

## When to Use

- {Scenario 1}
- {Scenario 2}

## Context Validation Checkpoints

- {Pre-condition question to validate before proceeding}
- {Another question}

## Steps

### 1. {Step Name}

{Description of what this step does and how to implement it.}

### 2. {Step Name}

{Description}
```

### Field Details

| Field | Required | Description |
|-------|----------|-------------|
| Name (H1) | Yes | Human-readable command name |
| Description | Yes | What the command does. First sentence used as summary in agent config. |
| When to Use | Yes | Scenarios where this command is appropriate |
| Context Validation | Recommended | Questions to answer before running the command |
| Steps | Yes | Numbered steps with sub-instructions |

## Deployment Rules

When the `sync` workflow deploys from `.codeplaybook/` to an agent's directory, the agent adapter determines the output format:

| Agent | Standards Deploy To | Commands Deploy To | Frontmatter |
|-------|--------------------|--------------------|-------------|
| Claude Code | `.claude/rules/codeplaybook-{slug}.md` | `.claude/commands/codeplaybook-{slug}.md` | `description`, `globs` (from Scope), `alwaysApply` |
| Cursor | `.cursor/rules/codeplaybook-{slug}.md` | `.cursor/commands/codeplaybook-{slug}.md` | `globs`, `alwaysApply` |
| GitHub Copilot | `.github/instructions/codeplaybook-{slug}.md` | `.github/prompts/codeplaybook-{slug}.prompt.md` | `applyTo` |

### Scope → Glob Conversion

Agent adapters convert the human-readable `## Scope` section into agent-specific file-matching patterns:

| Scope Description | Glob Pattern |
|-------------------|-------------|
| "TypeScript files" | `**/*.ts` |
| "React components" | `**/*.tsx` |
| "All test files" | `**/*.{test,spec}.{ts,js,tsx,jsx}` |
| "Python files in src/" | `src/**/*.py` |
| Too broad or unclear | Omit glob (apply everywhere) |

## Portability

The `.codeplaybook/` directory is designed to be:
- **Version controlled** — commit it to your repository
- **Shared** — copy it to other projects that follow the same standards
- **Agent-agnostic** — no agent-specific syntax in the source files
- **Human-editable** — plain markdown, no special tooling required to modify
